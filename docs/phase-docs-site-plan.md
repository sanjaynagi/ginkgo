# Documentation Site Plan

## Problem Definition

Ginkgo has useful raw documentation material in the README, architecture notes,
and runnable examples, but it does not yet provide a coherent end-user
documentation site. New users currently need to piece together installation,
workflow authoring, CLI usage, caching behavior, and notebook support from
multiple sources.

## Proposed Solution

Build a local-first Sphinx documentation site powered by MyST Markdown and a
curated API reference. The published content should target end users first,
anchor the narrative around the bioinformatics example workflow, and explain
architecture only where it helps users understand runtime behavior.

The documentation site will include:

- a modern landing page with a peach-toned visual identity inspired by ginkgo
  leaves
- getting-started guides for installation and first workflow execution
- concept guides for tasks, flows, environments, notebooks, caching,
  provenance, CLI usage, and the local UI
- one canonical bioinformatics walkthrough plus a focused notebook example
- a minimal API reference for the public user-facing surface

## Risks And Tradeoffs

- A broad autogenerated API reference would expose thin or uneven docstrings, so
  the first phase should keep the reference intentionally small.
- The bioinformatics example is the best canonical workflow today, but notebook
  support is demonstrated more fully in the retail example. The docs should keep
  the main narrative centered on bioinformatics and use the retail notebook only
  for the notebook guide.
- Existing internal planning documents contain stale details and should not be
  published directly into the user docs site.

## Success Criteria

- `pixi` can build the documentation site locally in a reproducible way.
- The site has clear navigation and an intentional visual identity.
- A new user can install Ginkgo, run the canonical example, understand the core
  execution model, and discover the main CLI and UI features without reading
  source code.
- The docs structure is ready for future expansion without needing a major
  reorganization.
